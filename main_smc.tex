\begin{abstract}
Retrieval-augmented generation (RAG) is a strong baseline for evidence-grounded QA, but tutoring quality depends on more than retrieval: learner diagnosis, scaffold choice, rubric alignment, and calibrated help. We present a repository-grounded adaptive tutoring system that routes each example among classical RAG, hybrid agentic RAG, and coordination-first non-RAG pipelines. The implementation combines lightweight regime routing, specialist-agent orchestration, conditional retrieval, and profile-aware evaluation. Using frozen benchmark snapshots from EduBench and TutorEval, we find regime-dependent winners rather than a single dominant architecture. On fully converged TutorEval runs, the hybrid route gives the best quality-efficiency frontier (Token-F1, grounded overlap, rubric coverage, and latency). On partial EduBench trajectories, coordination-heavy routes currently lead rubric-oriented metrics while retrieval-heavy routes remain strongest on grounding-oriented signals. The evidence supports a systems conclusion: in tutoring, retrieval is most effective as a conditional tool inside an adaptive coordination policy.
\end{abstract}

\section{Introduction}
Classical RAG improved factual grounding in language-model pipelines \cite{lewis2020rag}, but tutoring workloads require additional control dimensions: when to explain, when to withhold, how to scaffold, and how to stay aligned with a grading rubric. Recent educational benchmarks emphasize these dimensions directly, including pedagogical quality, adaptivity, and safety \cite{xu2025edubench,chevalier2024sciencetutors,macina2025mathtutorbench,srinivasa2025tutorbench}. In parallel, multi-agent studies suggest that once tasks require planning and staged coordination, orchestration policy can dominate prompt-only gains \cite{nguyen2025marag,radeva2025coordination}.

We study this gap with a repository-grounded question: \emph{when should tutoring systems treat retrieval as the main controller, and when should retrieval be a conditional component inside a broader multi-agent policy?} Our contribution is an implementation-backed comparison across architecture families already executable in the codebase: \texttt{classical\_rag}, \texttt{agentic\_rag}, \texttt{non\_rag\_multi\_agent}, \texttt{single\_agent\_no\_rag}, and \texttt{hybrid\_fast}. We report both quality and operating-cost behavior under the frozen session snapshot.

\section{Methodology}
\subsection{Adaptive routing and execution policy}
The runtime uses a \texttt{LightweightRegimeRouter} to map each example to a regime and architecture family. Routing combines dataset priors with feature scores for evidence demand, coordination demand, rubric demand, planning demand, and tutoring demand. The router emits execution flags (for example, \texttt{require\_retrieval}, \texttt{use\_rubric\_agent}, \texttt{use\_critic}) so control policy is explicit rather than hidden in prompts.

Pipeline execution is architecture-specific but interface-consistent. In agentic and hybrid paths, planner, diagnoser, and rubric specialists run in parallel before tutor synthesis. The hybrid path keeps retrieval conditional: retrieval is skipped when route-level evidence demand is low, and a fallback retrieval pass is triggered only when initial retrieval is empty and evidence score remains high enough for re-query.

\subsection{Conditional retrieval stack}
Retrieval uses a lightweight hybrid index with weighted lexical, character, and latent signals:
\begin{equation}
s(q,d)=\alpha s_{\text{lex}}+\beta s_{\text{char}}+\gamma s_{\text{latent}},
\end{equation}
with default weights exposed in configuration for reproducible ablation. Candidate chunks are deduplicated, reranked with compact pair features (base score, overlap features, phrase and numeric matches, brevity), and packed by MMR-style context selection under a strict character budget.

\subsection{Evaluation protocol}
Evaluation is profile-aware at both adapter and metric layers. For EduBench, the adapter builds a consensus-style supervision profile from model predictions, score fields, and rubric terms. For TutorEval, key points are converted into rubric targets and used as a surrogate supervision signal when explicit gold text is absent. Metrics are computed per example and aggregated across runs, with trace logging for architecture choice, retrieval behavior, and latency.

\section{Experimental setup}
We use the frozen session snapshot from repository artifacts (timestamped 2026-04-12 21:18:44 EDT). Two datasets are included: EduBench (large-scale tutoring benchmark; run still in progress at freeze time) and TutorEval (all four architectures fully complete at freeze time). This distinction is critical for interpretation: TutorEval supports endpoint comparison, while EduBench supports trajectory comparison only.

We report correctness/overlap, rubric coverage, grounding-oriented overlap, profile-specific alignment metrics, and operating-cost metrics (latency and generated tokens). This paper treats those axes separately to avoid collapsing pedagogical quality and systems efficiency into a single scalar.

\section{Results}
Table~\ref{tab:frozen_quality_summary} reports the frozen quality-efficiency summary. The strongest defensible endpoint result comes from TutorEval (all architectures complete). On TutorEval, \texttt{hybrid\_fast} is the best quality-efficiency operating point: it achieves best token-level overlap, best grounded overlap, best rubric coverage, best keypoint recall, and lowest latency/tokens among compared architectures.

On EduBench, we avoid endpoint ranking claims because all runs are partial at freeze time. Still, the trajectory shows a consistent pattern: coordination-heavy routes currently lead rubric-oriented metrics and efficiency, while retrieval-heavy routes remain stronger on grounding-oriented overlap and score-alignment fields in this profile. This regime-dependent split is exactly the behavior expected from conditional retrieval rather than fixed retrieval-first control.

\section{Discussion and limitations}
These findings support a practical design rule for tutoring systems: choose architecture by regime, not by doctrine. Retrieval-centric control is preferable when evidence grounding is the dominant bottleneck, while coordination-first control improves rubric alignment and efficiency when pedagogical orchestration dominates.

The current evidence has clear limits. First, EduBench comparisons are partial-run trajectories, not fully converged endpoints. Second, EduBench and TutorEval expose different supervision signals and profile-specific metric availability, so cross-dataset averages should not be over-interpreted. Third, some metrics contain construction-dependent behavior (for example, retrieval recall dependence on expected document IDs), which requires careful reading of protocol details.

\section{Conclusion}
We presented an implementation-grounded comparison of retrieval-first and coordination-first tutoring architectures under a shared adaptive router. The key result is regime dependence: no single architecture dominates across educational workloads. Fully converged TutorEval results favor hybrid conditional retrieval, while partial EduBench trajectories indicate stronger rubric-oriented behavior from coordination-heavy routes. The broader implication is architectural: in tutoring, retrieval should be orchestrated as a conditional capability within an adaptive multi-agent policy.
