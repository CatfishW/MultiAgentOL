\begin{abstract}
Large language models can generate fluent responses, but reliable domain assistance requires grounding in external evidence, visible user or task state, and explicit answer criteria. Classical retrieval-augmented generation (RAG) usually treats grounding as a fixed retrieve-then-read step in which a query is matched against a corpus and the generator reads the returned passages. This control flow is rigid when the same surface question requires different evidence, response structure, or evaluation criteria across situations. We propose \textsc{MARLET} (Multi-Agent Retrieval with Lightweight Evidence and Task-State Control), a configurable multi-agent retrieval framework that treats RAG as structured prompt construction rather than query-only passage lookup. A supervisory router identifies the response regime; LLM-backed planning, state diagnosis, criteria analysis, and conditional retrieval prepare typed context blocks; and all intermediate outputs are reassembled into one augmented prompt for a fixed generator and bounded critic. The reusable mechanism is domain-general: only cue lexicons, default criteria, and corpus/index contents are domain configuration. Controlled validation in educational tutoring under matched backbone, corpus, reranker, and response cap shows that \textsc{MARLET} improves quality proxies while reducing token use, with additional latency and model-call reductions in some settings.
\end{abstract}

\section{Introduction}
Large language models (LLMs) are increasingly used as domain assistants because they can generate explanations, diagnose errors, and provide feedback. However, a useful assistant cannot rely only on parametric knowledge. Knowledge-grounded dialogue has long shown the value of external evidence~\cite{dinan2019wizard}; domain assistance similarly needs grounding in manuals, policies, evaluation criteria, worked examples, user profiles, or task records to avoid fluent but misaligned guidance. Educational tutoring is a demanding instantiation of this broader problem~\cite{chevalier2024sciencetutors,liu2026agenttutor}.

Classical retrieval-augmented generation (RAG) has therefore become the default grounding mechanism~\cite{lewis2020rag}. In classical RAG, the query is embedded, the top-$k$ most similar passages are retrieved, and the LLM generates a response conditioned on those passages. Thus, relevance depends heavily on retrieval quality and query representation~\cite{thakur2021beir}.

Domain assistance is more complex than factual question answering. A user's question is often only a partial signal: the appropriate response may depend on expertise, goal, task history, constraints, prior confusion, and evaluation criteria. We call these visible attributes the user's \emph{personalized task state}. In education, this state becomes learner level and misconceptions; in technical support, it becomes device state and attempted fixes; in policy or legal assistance, it becomes jurisdiction, role, and applicable rules. Therefore, retrieval must find evidence that is appropriate for the task state, not merely topical.

Classical RAG can include explicit state text when supplied, but appending user state, dialogue, criteria, and task constraints forces one query embedding to compress topic, level of detail, intent, and response requirements. As constraints grow, search intent can blur. The issue is therefore how a response system should organize constraints before retrieval and generation.

Therefore, we introduce \textsc{MARLET} (Multi-Agent Retrieval with Lightweight Evidence and Task-State Control). Although instantiated for educational validation, MARLET is a configurable RAG controller: instead of letting the retriever act as the top-level controller, a supervisory router identifies the response regime and decides whether retrieval is needed. A planner scopes search intent, a state diagnoser summarizes visible user or task state, and a criteria module states answer requirements; these modules run in parallel when possible, then are reassembled with retrieved evidence into one generator prompt. The framework integrates selective retrieval, tool-using agents, and state-aware criteria modeling into one context-construction mechanism.

The experiments validate the framework rather than present a model leaderboard. We compare with classical retrieve-then-read RAG and Agentic RAG under shared backbone, corpus, reranker, response cap, generator, and critic. The educational instantiation shows that the same generator scores higher when fed context assembled by \textsc{MARLET}, with lower resource cost in several settings.

\noindent\textbf{Contributions.} (i) We propose \textsc{MARLET}, a configurable multi-agent retrieval framework with a supervisory router, planner, state diagnoser, criteria module, conditional retriever, generator, and critic. (ii) We formalize the router and retrieval gate as transparent, training-free controls over task state, answer criteria, and external evidence. (iii) We instantiate and validate the framework against classical RAG and Agentic RAG baselines on two educational benchmarks under shared backbone, corpus, reranker, and response cap.

\section{Related Work}

Classical RAG fixes retrieval as the first controller: the query is matched to passages, and generation conditions on the returned evidence~\cite{lewis2020rag}. Selective and corrective RAG methods improve this controller. Self-RAG learns when to retrieve, generate, and critique; CRAG evaluates retrieved documents and attempts correction when retrieval is unreliable; Adaptive-RAG and TARG vary retrieval behavior by query complexity or retrieval utility~\cite{asai2024selfrag,yan2024crag,jeong2024adaptiverag,wang2026targ}. RAG diagnostics further emphasize separating retrieval, ranking, and generation errors~\cite{ru2024ragchecker}. These methods address whether passages retrieved for a query are necessary, sufficient, or reliable, but they do not explicitly organize user/task state, criteria, and evidence as separate prompt constraints before retrieval.

Agentic RAG and multi-agent LLM systems move beyond a single retrieve-then-read step. ReAct, AutoGen, CAMEL, and MetaGPT show that tool use and role decomposition can improve complex task execution~\cite{yao2023react,wu2024autogen,li2023camel,hong2024metagpt}. MA-RAG uses multiple agents for query disambiguation, evidence extraction, and answer synthesis on ambiguous or multi-hop QA tasks~\cite{nguyen2025marag}. These systems demonstrate coordination benefits, but their roles are usually tied to task execution or ambiguous question answering rather than a reusable controller that chooses among planning, criteria checking, state adaptation, and evidence grounding before final prompt assembly.

Recent educational work shows one high-stakes setting where state-aware grounding matters. AgentTutor builds multi-turn personalized learning with learner assessment and memory~\cite{liu2026agenttutor}; adaptivity studies show that LLM tutors are sensitive to omitted student-context features~\cite{borchers2025adaptivity}; and AI-tutor evaluation emphasizes pedagogy, mistake diagnosis, guidance quality, and learner support~\cite{maurya2025aitutoreval,chevalier2024sciencetutors,xu2025edubench,macina2025mathtutorbench,srinivasa2025tutorbench}. These works motivate education as a validation domain, but they do not formulate a domain-configurable retrieval controller for deciding when to retrieve, what evidence to retrieve, and how much prompt space to allocate to evidence versus state and criteria.

\section{Methodology}
\label{sec:methodology}

\textsc{MARLET} turns one domain request into a grounded response. It is a RAG controller rather than a set of independent answer agents: every intermediate output is converted into context for one final LLM prompt. Its design integrates three domain-general principles: route before retrieval, exchange typed records rather than free-form chat, and retrieve only when the response brief needs external evidence. In education, the final generator is a tutor; in another domain, the same controller could drive a legal assistant, clinical summarizer, or technical support agent by replacing cue lexicons, default criteria, and the corpus. The flow is unchanged: standardize the entry, score routing cues, run active LLM-backed specialists in parallel, merge their outputs with retrieved evidence, serialize one augmented prompt, and run one critic revision if needed.

\subsection{Input Entry and Inference Caps}

For one sample, \textsc{MARLET} first standardizes all visible fields into
\begin{equation}
x=(q,o,c,h,r,v).
\end{equation}
Here, $q$ is the user question or task prompt; $o$ is an optional answer-option list; $c$ is optional inline context such as a policy excerpt, manual page, or chapter passage; $h$ is recent interaction history; $r$ is an optional criteria list; and $v$ is an optional image list. Empty fields are allowed, so the same representation covers factual questions, assessment requests, feedback requests, and planning requests.

The only user-facing output is the final answer $y$. Internally, \textsc{MARLET} keeps a route record, an evidence bundle $D$ with packed retrieved chunks, and citation identifiers linking chunks in $D$ to their sources. Planner output, state summary, criteria summary, and $D$ are assembled as prompt blocks before the LLM generator produces $y$. Fixed inference caps keep the process bounded: $K$ caps LLM planner and retrieval-query outputs, $Q$ caps packed chunks/tool calls, $L$ caps packed-evidence length, and $N$ caps response tokens.

The core design choice is to avoid asking one retrieval query to carry every constraint. \textsc{MARLET} instead fills three typed context slots before generation. The \emph{planner} writes $p=(a,P)$: $a$ is a short operating strategy, and $P$ is a capped list of searchable intents. This slot answers, ``what should the system try to do, and what evidence would be useful?'' The \emph{state diagnoser} writes $\ell$, a summary of only visible task state from $q$ and $h$, such as expertise level, goal, confusion, prior attempts, or requested style; it is not a persistent profile and does not infer hidden demographics. The \emph{criteria analyst} writes $u$, the answer requirements: it preserves explicit criteria $r$ when available and otherwise supplies a small checklist for correctness, clarity, evidence use when available, and actionable next steps. This separation follows the practical observation that search intent, task state, and answer criteria affect different parts of the response and should not be compressed into one embedding string~\cite{liu2026agenttutor,borchers2025adaptivity}.

\subsection{Supervisor Routing}

The supervisor begins with a routing blob:
\begin{equation}
b(x)=\mathrm{concat}(q,c,r,h),
\end{equation}
where $\mathrm{concat}(\cdot)$ means that the question, inline context, criteria text, and interaction turns are joined into one normalized string. Images remain available to the final generator, but they are not converted into the textual routing score.

The router uses four cue families $j\in\{\mathrm{ev},\mathrm{cr},\mathrm{pl},\mathrm{ad}\}$ for evidence, criteria, planning, and adaptation. These are generic response-construction needs rather than benchmark or domain labels. Evidence cues ask for source-grounded support; criteria cues ask the system to judge an answer against explicit requirements; planning cues ask for an ordered action path; and adaptation cues indicate that the response should be shaped by user state, expertise, or prior interaction. Each family has a keyword set $W_j$. Evidence cues include \emph{source}, \emph{document}, \emph{manual}, and \emph{verification}; criteria cues include \emph{score}, \emph{criteria}, and \emph{comment}; planning cues include \emph{plan}, \emph{sequence}, and \emph{next steps}; adaptation cues include \emph{user}, \emph{beginner}, \emph{expert}, \emph{confused}, and \emph{explain}. Let $M_j(x)$ be the number of cues from $W_j$ that appear in $b(x)$. The cue score is
\begin{equation}
s_j(x)=\frac{M_j(x)}{|W_j|},
\end{equation}
where $|W_j|$ is the number of cues in family $j$, so $s_j(x)$ is the fraction of that cue family observed in the routing blob. The thresholds are fixed router hyperparameters, not probabilities or learned weights. We choose them in cue-count units so $\tau_{\rm mid}$ requires a repeated cue pattern rather than one accidental word, $\tau_{\rm plan}$ is stricter because a false planning route changes the whole answer structure, and $\tau_2$ permits retry only for strong evidence cues. Porting the framework to a new domain mainly means replacing $W_j$ and default criteria while keeping the router equation and control flow.

After cue scoring, the supervisor chooses a response regime. A regime is not a dataset label; it is a generic prompt-priority decision made before retrieval and generation. The intuition is to ask what missing constraint would most likely make the final answer wrong. If the user needs an ordered path, planning (PL) reserves space for sequence and dependencies. If the task asks for judgment, criteria feedback (CF) reserves space for evaluation rules and scoring logic. If the user's state changes how the same content should be presented, adaptive response (AR) reserves space for user state. If correctness depends mainly on outside material, evidence-grounded response (EG) reserves space for source evidence. Because these needs can overlap, $R(x)$ is selected by the priority rule
\begin{equation}
R(x)=
\begin{cases}
\mathrm{PL}, & s_{\rm pl}\ge s_{\rm cr},\ s_{\rm pl}\ge s_{\rm ad},\ s_{\rm pl}\ge\tau_{\rm plan},\\
\mathrm{CF}, & s_{\rm cr}\ge s_{\rm ad},\ s_{\rm cr}\ge\tau_{\rm mid},\\
\mathrm{AR}, & s_{\rm ad}\ge \tau_{\rm mid},\\
\mathrm{EG}, & \mathrm{otherwise}.
\end{cases}
\end{equation}
The order reflects generic constraint strength. Sequence changes the whole response, explicit criteria constrain evaluation, user state shapes delivery, and evidence is the default grounding route when the request does not clearly require one of the first three prompt shapes.

Next, the supervisor uses a \emph{retrieval gate}: a binary control for whether external evidence enters the final prompt. Inspired by selective-retrieval research, MARLET integrates this idea as a prompt-allocation control that opens only when outside grounding is needed~\cite{asai2024selfrag,jeong2024adaptiverag,wang2026targ}:
\begin{equation}
G(x)=
\begin{cases}
1, & s_{\rm ev}\ge \tau_{\rm mid}\ \mathrm{or}\ R(x)=\mathrm{EG},\\
0, & \mathrm{otherwise}.
\end{cases}
\end{equation}
Thus $G(x)=1$ runs retrieval; $G(x)=0$ builds the same prompt without external chunks. A stricter retry gate runs one raw-question pass only when the first bundle is empty:
\begin{equation}
G_2(x)=
\begin{cases}
1, & D_0\ \mathrm{is\ empty\ and}\ s_{\rm ev}\ge \tau_2,\\
0, & \mathrm{otherwise}.
\end{cases}
\end{equation}
where $D_0$ is the evidence bundle from the first retrieval attempt.

\subsection{Specialist Context Agents and Coordination}

After routing, MARLET knows which prompt priority must be protected: sequence, criteria, adaptation, or evidence. It then builds a response brief through LLM-backed specialist agents inspired by tool-using and multi-agent systems~\cite{yao2023react,wu2024autogen}. These agents are not independent tutors and do not produce competing answers. They are brief writers: each receives the standardized entry and route decision, returns a JSON-like typed record, and stops. The coordinator then assembles the records into one shared context.

The three records have different downstream uses. The planner's $P$ is the only specialist output consumed by retrieval; it converts the original request into compact search intents, which prevents long dialogue, criteria text, or user-state text from becoming retrieval noise. The state summary $\ell$ is used by the generator and critic to match explanation depth and response style to visible conditions. The criteria summary $u$ is used by the generator and critic to keep the answer aligned with explicit or default requirements. When no explicit criteria are present and the route is not criteria feedback, $u$ is filled by the same default checklist rather than by an extra LLM call.

This is also the communication pattern among agents. MARLET uses fan-out/fan-in coordination: the coordinator sends the same normalized entry to active specialists in parallel, waits for their typed outputs, fills deterministic fallbacks if any output is malformed, and merges $(R,G,p,\ell,u)$ into one response brief. If $G=1$, retrieval uses $P$ to build evidence bundle $D$; if $G=0$, the brief bypasses external chunks. In both cases, the generator receives explicit prompt blocks for plan, visible state, criteria, and evidence instead of inferring these constraints from one overloaded query.

\subsection{Conditional Retrieval and Evidence Packing}

\textsc{MARLET} contains retrieval, but retrieval is not the top-level controller and does not require a traditional dense vector database. When $G(x)=1$, an LLM-backed retrieval agent refines the planner's search intents into up to $K$ compact queries, and each query $q_i$ searches a local hybrid text index. For a query $q_i$ and candidate chunk $d$, the base retrieval score is
\begin{equation}
\mathrm{base}(q_i,d)=
w_{\rm lex}\mathrm{lex}+w_{\rm char}\mathrm{char}+w_{\rm svd}\mathrm{svd}+\beta .
\end{equation}
Here $\mathrm{lex}$ is word TF--IDF similarity, $\mathrm{char}$ is character $3$--$5$ gram TF--IDF similarity, $\mathrm{svd}$ is truncated-SVD similarity, $\beta$ is a small title/source boost, and the $w$ terms are fixed retriever weights. Candidates from all planner queries are merged, and duplicate chunks keep their highest score.

The reranker then scores each candidate against the original question:
\begin{equation}
\begin{aligned}
\mathrm{rank}(q,d)=&\alpha_{\rm base}\mathrm{base}+\alpha_{\rm tok}o_{\rm tok}
+\alpha_{\rm title}o_{\rm title}\\
&+\alpha_{\rm phr}m_{\rm phr}+\alpha_{\rm num}o_{\rm num}
+\alpha_{\rm len}b_{\rm len}.
\end{aligned}
\end{equation}
Here $o_{\rm tok}$ is token overlap with $q$, $o_{\rm title}$ is title-token overlap, $m_{\rm phr}$ is an exact phrase-match flag, $o_{\rm num}$ is numeric overlap, $b_{\rm len}$ rewards moderate chunk length, and the $\alpha$ terms are fixed reranker weights. The state summary $\ell$ affects retrieval through the route and planner queries, not these weights.

Evidence packing integrates the relevance-diversity tradeoff used in MMR-style selection~\cite{carbonell1998mmr}: the context should be useful, but repeated chunks should not consume the limited evidence space. For each candidate chunk $d$, \textsc{MARLET} computes
\begin{equation}
\mathrm{pack}(d)=\lambda\,\mathrm{rank}(q,d)-(1-\lambda)\,\mathrm{repeat}(d).
\end{equation}
Here $\mathrm{repeat}(d)$ is the largest sentence overlap between $d$ and selected chunks, and $\lambda$ controls the relevance-diversity balance. The packer adds the remaining chunk with the highest $\mathrm{pack}(d)$ score until $Q$ chunks or the $L$ character limit is reached, forming evidence bundle $D$.

\subsection{Generation and Critic Revision}

This is where retrieval and agent outputs converge: $(R,G,p,\ell,u)$ supplies route, plan, state, and criteria fields, while $D$ supplies compact cited evidence. The retriever does not answer; its chunks become an evidence block. The final prompt contains task fields, response brief, evidence or inline context, and output requirements. The generator is the LLM-driven answer writer:
\begin{equation}
y_0=A_{\rm gen}(q,o,h,p,\ell,u,D,c).
\end{equation}
Its prompt contains $q$, optional answer options $o$, recent interaction history $h$, state summary $\ell$, plan $p$, criteria summary $u$, and either retrieved evidence $D$ or inline context $c$. The generator is instructed by the domain configuration to be correct, useful, concise but not terse, include a next step when appropriate, and cite retrieved evidence with document markers.

The critic performs one bounded revision pass. It checks for missing citation markers when $D$ is nonempty, uncovered criteria from $r$, or a response style that conflicts with the inferred state. If no issue is found, $y=y_0$; otherwise the critic rewrites once using $(y_0,q,h,p,\ell,u,D,r)$ and the revised $y$ becomes the final answer.

\section{Validation Experiments}
\label{sec:results}

We validate the framework on two public educational benchmarks used as domain-specific stress tests. TutorEval~\cite{chevalier2024sciencetutors} pairs science tutoring questions with long STEM chapters and teaching key points; EduBench~\cite{xu2025edubench} covers diverse educational scenarios with prompts, metadata, and reference model predictions that our adapter converts into score and rubric supervision.

The primary validation compares three context controllers under matched infrastructure, with a \textsc{Qwen3.5-27B-FP8} replication checking backbone sensitivity. Classical RAG retrieves first from the raw standardized prompt; visible learner profile, student answer, dialogue, and criteria text remain available to its generator prompt, so the baseline is not deprived of task context. Its limitation is controller order: retrieval happens before specialist summaries can shape search. Agentic RAG runs the same planner, state diagnoser, and criteria module before retrieval, but retrieval is always on. \textsc{MARLET} keeps the same specialists and retriever, then adds supervisory routing, conditional retrieval, and retry control. We report quality and cost together because only context construction changes.

\subsection{Implementation Details}

All controllers use the same deployed Python framework, corpus, hybrid text index, reranker, generator, critic, and metric code; only context assembly differs. The planner, state diagnoser, criteria module, retrieval-query refiner, generator, and critic use the same configured backbone; deterministic fallbacks run only when an LLM response is unavailable or malformed. LLM endpoints are served through SGLang, whose runtime is designed for structured multi-call LLM programs and efficient KV-cache reuse~\cite{zheng2024sglang}. Runs use \textsc{Qwen3.5-4B} and \textsc{Qwen3.5-27B-FP8}, reasoning budget 512, generation temperature 0.15, specialist JSON temperature 0.0, and response cap $N{=}900$ on \texttt{benwulab-Lambda-Vector} with two NVIDIA RTX 6000 Ada GPUs. We set $K{=}3$, $Q{=}4$, $L{=}6000$, router thresholds $(\tau_{\rm mid},\tau_{\rm plan},\tau_2)=(0.35,0.42,0.45)$, retriever weights $(0.58,0.17,0.25)$, reranker weights $(0.55,0.20,0.10,0.06,0.05,0.04)$, and packing weight $\lambda{=}0.68$.

Evaluation is dataset-aware, and all quality scores are automatic proxies rather than official benchmark or human grades. Inspired by RAG evaluation work~\cite{ru2024ragchecker}, we separate answer overlap, criteria fulfillment, grounding, and resource cost. Token-F1 balances reference-token coverage against extra generated tokens; rubric coverage checks whether benchmark criteria are expressed with exact or content-bearing partial matches; latency is wall-clock seconds per sample; and tokens count prompt plus completion tokens.

For EduBench, EB12D averages 12 normalized checks over scenario adaptation, factual/reasoning behavior, and pedagogical application, while EduAlign only measures extracted-score agreement with the benchmark consensus. For TutorEval, tutor-hit measures expert teaching-point coverage with full, partial, or zero credit.

\subsection{Trade-offs}

For TutorEval, the 27B paired CIs are positive for Token-F1, tutor-hit, rubric coverage, and latency. For EduBench, EB12D and rubric coverage improve, latency and token use decrease, and the small classical-RAG EduAlign edge is not statistically reliable.

\textsc{MARLET} can use more modules than classical RAG, so resource cost must be reported. Scoped planner queries and the retrieval gate reduce token use in 4B and still reduce latency in 27B, but 27B TutorEval shows that better coverage does not always mean fewer tokens. Thus \textsc{MARLET} decides what grounding a request needs; it is not a universal retrieval replacement.

\section{Limitations}
\label{sec:limitations}

The study validates a framework rather than a deployable assistant: evaluation is automatic and English-only, and user or task state is inferred from visible prompt/dialogue fields. Deployment requires privacy safeguards, bias checks, data governance, human oversight, and domain-specific outcome studies.

\section{Conclusion}
\label{sec:conclusion}

\textsc{MARLET} reframes grounding as supervised context construction rather than query-only passage lookup. With shared infrastructure, the educational instantiation improves fixed-generator quality and efficiency over classical RAG by coordinating task state, answer criteria, and conditional evidence before generation. Future work should test stronger controller baselines, additional domains, human and multi-turn outcomes, learned router calibration, profile-aware reranking, and privacy-preserving state inference.
